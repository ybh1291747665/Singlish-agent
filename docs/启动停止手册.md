# 启动停止手册

本文档提供本项目当前可运行 demo 的最小启动、验证与停止流程。

## 1. 前置条件

- 已安装并可用的 Docker Desktop
- 已创建并激活 `conda` 环境 `singlish-agent`
- 已执行过：

```powershell
python -m pip install -e apps/api
npm install
```

## 2. 启动顺序

### 2.1 启动基础设施

```powershell
docker compose -f infra/docker/docker-compose.yml up -d
```

检查状态：

```powershell
docker compose -f infra/docker/docker-compose.yml ps
```

期望至少看到：

- `singlish-agent-postgres`
- `singlish-agent-redis`
- `singlish-agent-minio`

### 2.2 初始化数据库表

当前项目默认不会在 API 启动时自动建表，所以真实联调前需要手动执行一次：

```powershell
@'
import asyncio
from singlish_agent_api.infrastructure.db.session import init_db

asyncio.run(init_db())
print("init_db ok")
'@ | python -
```

### 2.3 启动后端 API

```powershell
python -m uvicorn singlish_agent_api.main:app --app-dir apps/api/src --host 127.0.0.1 --port 8000
```

### 2.4 启动 Worker

```powershell
python -m singlish_agent_api.worker.run_worker
```

说明：
- 如果只是跑当前 demo，普通依赖即可。
- 如果要跑真实离线 ASR，Worker 必须在 `singlish-agent` conda 环境里启动，这样才能加载 `faster-whisper`、`av`、`torch` 等依赖。

### 2.5 启动前端

```powershell
npm run dev --workspace @singlish-agent/web -- --host 127.0.0.1 --port 5173
```

## 3. 快速验证

### 3.1 健康检查

```powershell
curl.exe http://127.0.0.1:8000/health
```

期望：

- `status` 为 `ok`
- `app`
- `database`
- `redis`
- `object_storage`

都为 `ok`

### 3.2 上传一个测试文件

```powershell
$sample = Join-Path $env:TEMP 'sample.wav'
Set-Content -Path $sample -Value 'demo-audio' -NoNewline
curl.exe -s -X POST -F "file=@$sample;type=audio/wav" http://127.0.0.1:8000/api/v1/jobs
```

期望返回：

- `job_id`
- `file_name`
- `status=queued`

### 3.3 查询任务状态

将上一步返回的 `job_id` 替换进去：

```powershell
curl.exe http://127.0.0.1:8000/api/v1/jobs/<job_id>
```

期望最终看到：

- `status=completed`
- `result_summary` 显示对应 provider 的成功信息
- `result_payload.transcription.provider` 为 `fake` 或 `faster_whisper`

### 3.4 查询时间戳分段

将上一步的 `job_id` 替换进去：

```powershell
curl.exe http://127.0.0.1:8000/api/v1/jobs/<job_id>/segments
```

期望：

- 返回 `job_id`
- 返回 `total_segments`
- `segments` 中包含 `start_seconds`、`end_seconds`、`text`

### 3.5 下载导出文件

将上一步的 `job_id` 替换进去：

```powershell
curl.exe -OJ http://127.0.0.1:8000/api/v1/jobs/<job_id>/exports/md
curl.exe -OJ http://127.0.0.1:8000/api/v1/jobs/<job_id>/exports/srt
```

期望：

- 能成功下载 Markdown 报告
- 能成功下载 SRT 字幕文件
- 返回头里包含 `Content-Disposition`

### 3.6 前端验证

在浏览器打开：

```text
http://127.0.0.1:5173
```

期望：

- Health 面板显示所有依赖为 `ok`
- 上传文件后可以看到 `job_id`
- 页面会轮询并最终显示 `completed`
- 页面会显示 Transcript Segments 时间轴
- 完成后会出现 Exports 下载区

## 4. 停止方式

### 4.1 停止前端 / API / Worker

如果它们是在当前终端前台运行，直接 `Ctrl+C`。

如果它们是单独后台进程，需要手动结束对应的 Python 或 Node 进程。

### 4.2 停止基础设施

```powershell
docker compose -f infra/docker/docker-compose.yml down
```

如果你还想同时清掉卷：

```powershell
docker compose -f infra/docker/docker-compose.yml down -v
```

注意：`-v` 会删除 PostgreSQL 和 MinIO 的本地卷数据。

## 5. 常见问题

### Docker CLI 可用，但无法连接 daemon

表现：

- `docker --version` 正常
- `docker compose up` 报 `//./pipe/docker_engine`

处理：

- 启动 Docker Desktop
- 等 `docker info` 正常返回后再重试

### PowerShell 的 `Invoke-RestMethod` 不支持 `-Form`

处理：

- 上传文件时优先用 `curl.exe -F`

### API 启动了，但 `/api/v1/jobs` 报数据库相关错误

处理：

- 确认 PostgreSQL 容器已健康
- 确认已执行一次 `init_db()`

### Windows 下 worker 消费任务后卡住或任务长期停留在 `queued`

处理：

- 确认 worker 是在 `singlish-agent` conda 环境里直接用 `python -m singlish_agent_api.worker.run_worker` 启动
- 当前代码已经在 worker 启动时切换到 Windows Selector event loop，避免 `asyncpg` 与默认 Proactor loop 的兼容问题
- 当前 worker 也会复用同一个 asyncio event loop，避免 `Future attached to a different loop` 这类跨 loop 连接错误
