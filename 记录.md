# 项目记录

本文档记录本项目开发过程中做出的重大决策、决策原因、遇到的问题，以及对应解决方法。

## 2026-06-10 到 2026-06-11 已完成决策

### 决策 1：采用 balanced monorepo 结构

- 决策内容：采用 `apps/api`、`apps/web`、`packages/contracts`、`packages/config`、`infra/docker` 的 monorepo 结构。
- 原因：
  - 需要同时承载后端 API、前端 demo、共享类型以及本地基础设施。
  - 比纯双应用仓库更利于后续共享 contract。
  - 比更重的 workspace 结构更轻，适合当前 demo-first 阶段。

### 决策 2：本地开发时只将基础设施放进 Docker

- 决策内容：PostgreSQL、Redis、MinIO 用 Docker Compose 运行；API、worker、web 直接本机运行。
- 原因：
  - 本机直接跑代码，调试更快。
  - 基础设施更适合容器化，便于统一版本和配置。
  - Windows 环境下如果把全部应用都放入容器，会显著增加开发和排查成本。

### 决策 3：优先完成“可运行的纵向切片”

- 决策内容：先完成健康检查、创建 job、读取 job、假处理 worker、前端上传和轮询，而不是提前实现 ASR 或 Agent 能力。
- 原因：
  - 先证明系统骨架可运行，比先堆空目录或未来能力更有价值。
  - 能更早暴露数据库、队列、对象存储、前后端契约方面的问题。

### 决策 4：测试在无本地基础设施时允许使用最小替身

- 决策内容：在本机未运行 PostgreSQL / Redis / MinIO 时，Task 6 和 Task 7 的部分测试使用 fake session / fake storage / fake queue。
- 原因：
  - 当时环境里没有 Docker 可用，无法进行真实依赖联调。
  - 仍需保证计划中的 API 和 worker 切片可以被验证。
  - 限制替身只用于测试层，生产代码保持真实边界。

### 决策 5：保持应用默认启动不强依赖数据库初始化

- 决策内容：通过 `init_db_on_startup` 控制初始化，默认值为 `False`。
- 原因：
  - 避免 API 在没有 PostgreSQL 时完全无法启动或无法跑测试。
  - 保持健康检查和非 DB 依赖测试可以独立运行。

### 决策 6：将对象存储与队列边界接入真实实现

- 决策内容：
  - `jobs` 路由从 noop storage / noop queue 切换为 `ObjectStorageService` 和 Celery 队列。
  - worker 使用 Celery 任务 `singlish_agent.process_job` 驱动假处理流程。
- 原因：
  - 这一步是从“纯 demo stub”走向“真实系统骨架”的关键。
  - 后续真实音频处理可以继续沿这两个边界演进，不需要重写调用链。

### 决策 7：前端使用最小 Vite + React + Vitest 工作区

- 决策内容：建立 `apps/web` workspace，并创建 `packages/contracts` 共享前后端 TypeScript 类型。
- 原因：
  - 快速搭建可运行 demo UI。
  - 将健康响应、job 创建响应、job 详情响应统一收敛到 contracts。

## 已遇到的问题与解决方法

### 问题 1：最初环境中 Docker 不可用

- 表现：
  - 早期无法运行 `docker compose`。
  - 无法进行真实基础设施联调。
- 解决方法：
  - 先按计划完成代码骨架和测试。
  - 用 fake session / fake infra 完成受限环境下的验证。
  - 待用户安装 Docker 后，再继续进行真实联调。

### 问题 2：API 默认启动时会尝试初始化数据库，导致无 PostgreSQL 时启动失败

- 表现：
  - FastAPI lifespan 中直接执行 `init_db()`。
- 解决方法：
  - 加入 `init_db_on_startup` 配置。
  - 默认不初始化数据库，只在显式开启时执行。

### 问题 3：健康检查中的对象存储探测延迟过高

- 表现：
  - S3 `head_bucket` 在依赖不可达时耗时很长。
- 解决方法：
  - 使用较短的 boto 超时配置。
  - 将阻塞型 S3 调用放入线程，避免卡住事件循环。

### 问题 4：真实 worker 启动时没有注册 `process_job` 任务

- 表现：
  - Celery worker 入口只加载了 `celery_app`，未显式包含任务模块。
  - 真实 worker 运行时可能看不到 `singlish_agent.process_job`。
- 解决方法：
  - 在 Celery app 中加入 `include=["singlish_agent_api.worker.tasks"]`。
  - 新增测试验证任务已经注册到 `celery_app.tasks`。

### 问题 5：job 状态在外部副作用成功前被提前写成成功态

- 表现：
  - 上传失败时 job 可能已被写成 `uploaded`。
  - 入队失败时 job 可能已被写成 `queued`。
- 解决方法：
  - 调整顺序：
    - 上传成功后再转到 `uploaded`
    - 队列发布成功后再转到 `queued`
  - 为上传失败和入队失败分别补测试。

### 问题 6：worker 出错后 job 会卡在 `processing`

- 表现：
  - 假处理任务执行中发生异常时，job 不会进入终态。
- 解决方法：
  - 在 worker 中捕获处理阶段异常。
  - 当状态已进入 `processing` 时，将 job 迁移到 `failed` 后再抛出异常。
  - 增加失败路径测试。

### 问题 7：前端构建因测试回调类型不匹配而失败

- 表现：
  - `UploadPanel.test.tsx` 中的 `onUpload` mock 返回了 `Promise<number>`，与组件要求的 `Promise<void>` 不匹配。
- 解决方法：
  - 改成显式返回 `void` 的异步函数。
  - 重新运行前端测试和构建，确认通过。

## 2026-06-11 联调阶段新增问题与解决方法

### 问题 8：Docker CLI 可用，但 daemon 未运行

- 表现：
  - `docker --version` 正常。
  - `docker compose up` 报错：无法连接 `//./pipe/docker_engine`。
- 解决方法：
  - 启动 Docker Desktop。
  - 等待 `docker info` 可正常返回后，再执行 `docker compose up -d`。

### 问题 9：并行执行 `docker compose up` 与 `docker compose ps` 时，`ps` 输出为空

- 表现：
  - 首次拉镜像耗时较长。
  - 与之并行执行的 `ps` 在容器真正创建前就返回了空列表。
- 解决方法：
  - 在 `up -d` 完成后单独再次执行 `docker compose ps`。
  - 使用 `docker ps` 辅助核对健康状态与端口映射。

### 问题 10：PowerShell 的 `Invoke-RestMethod` 不支持 `-Form`

- 表现：
  - 使用 `Invoke-RestMethod -Form` 上传文件时报参数不存在。
- 解决方法：
  - 改用 `curl.exe -F` 发送 multipart 表单。
  - 继续验证真实的 `/api/v1/jobs` 上传链路。

### 问题 11：Windows 下 `Start-Process` 不能直接用 `npm`

- 表现：
  - 启动 Vite 开发服务器时，`Start-Process -FilePath npm` 报 `%1 is not a valid Win32 application`。
- 解决方法：
  - 改用 `C:\Program Files\nodejs\npm.cmd` 作为可执行文件。
  - 随后成功启动前端开发服务器，并确认 `http://127.0.0.1:5173` 返回 `200`。

## 2026-06-11 下一阶段 A 问题与解决方法

### 问题 12：旧 worker 失败测试仍然绑定旧的 `processing` / `normalizing` 假设

- 表现：
  - 当 worker 阶段扩展后，原失败测试的触发点不再正确
  - 导致测试没有真正打到新的失败路径
- 解决方法：
  - 将失败注入点调整到 `generating_report` 阶段
  - 让测试继续覆盖“处理中失败后转为 `failed`”这一真实意图

### 决策 15：用单个结构化结果载荷承载各阶段产物

- 决策内容：
  - 在 job 上增加 `result_payload`
  - 由一个结构化对象统一保存各阶段假结果：
    - `preprocessing`
    - `transcription`
    - `normalization`
    - `report`
- 原因：
  - 比为每个阶段单独加多列更轻
  - 更适合当前快速演进阶段
  - 后续接真实 ASR / 标准化 / 报告时，只需替换阶段生成逻辑，不必重塑 API 形状

### 决策 16：在没有完整迁移系统前，采用兼容式建列

- 决策内容：
  - 在 `init_db()` 中加入 `ALTER TABLE jobs ADD COLUMN IF NOT EXISTS result_payload TEXT`
- 原因：
  - 当前项目还没有 Alembic 迁移流
  - 仅靠 `create_all()` 无法给已存在的 `jobs` 表补新列
  - 兼容式建列能让现有本地联调环境继续工作，不必手动删库重建

### 问题 13：状态骨架完成后，系统仍然缺少可被前端消费的阶段产物

- 表现：
  - 虽然 worker 已经能走完整阶段链
  - 但 API 仍然只返回 `result_summary`
  - 前端无法展示 transcript、standard English、summary 等更真实内容
- 解决方法：
  - 增加 `result_payload`
  - worker 在每个阶段填充对应的假结果
  - API 返回解析后的结构化对象
  - 前端新增结果详情组件展示这些中间与最终产物

### 问题 14：现有 PostgreSQL 表结构无法自动获得新字段

- 表现：
  - 数据模型增加 `result_payload` 后，已存在的本地 `jobs` 表不会自动更新
- 解决方法：
  - 显式运行 `init_db()`
  - 由 `ALTER TABLE ... ADD COLUMN IF NOT EXISTS ...` 自动补齐新列
  - 重启 API 与 worker 后再进行真实联调验证

## 2026-06-11 下一阶段 A 真实 ASR 接入决策

### 决策 17：先接入可懒加载的真实 ASR provider，并保留自动回退

- 决策内容：
  - 新增 `infrastructure/asr/provider.py`
  - 将转写能力抽象成 provider
  - 默认使用 `auto` 策略：优先尝试 `faster_whisper`，失败时自动回退到 `fake`
- 原因：
  - 这样可以在不破坏现有 demo 稳定性的前提下，逐步把假链路替换成真链路
  - 本地不同环境的依赖完整度不一致，自动回退能减少“因为缺包导致整条任务失败”的摩擦

### 决策 18：真实音频仍沿用对象存储入口，由 worker 下载后落临时文件再转写

- 决策内容：
  - worker 从 MinIO 下载上传音频
  - 写入临时文件后交给 ASR provider 处理
  - 处理结束后删除临时文件
- 原因：
  - `faster-whisper` 对文件路径输入最直接
  - 这条链路更贴近未来真实处理流程，而不是只在内存里拼演示数据

### 问题 15：基础 Python 环境缺少真实 ASR 依赖

- 表现：
  - 当前正在运行的基础 Python 环境没有 `faster_whisper`、`torch`、`av` 等包
  - 直接在这个环境里启动 worker 时，只能走回退逻辑，无法真正做离线 ASR
- 解决方法：
  - 确认 `singlish-agent` conda 环境中已经安装这些依赖
  - 在该环境里执行 `python -m pip install -e apps/api`
  - 后续将 worker 切换到 `singlish-agent` 环境运行

### 问题 16：结果摘要仍写死为 fake 文案，会误导联调判断

- 表现：
  - 即使真实 provider 生效，`result_summary` 依然显示 fake 完成信息
- 解决方法：
  - 改为按 `transcription.provider` 动态生成摘要
  - 这样 API 与前端都能更清楚地区分当前到底命中了哪条链路

### 问题 17：真实 provider 本身可用，但运行中的 worker 仍可能停留在错误环境

- 表现：
  - 直接验证 provider 时，`singlish-agent` 环境已经可以产出 `faster_whisper` 结果
  - 但如果后台 worker 仍由基础 Python 环境启动，联调时仍只会看到回退效果
- 解决方法：
  - 将 API 与 worker 都切换到 `singlish-agent` conda 环境启动
  - 用真实语音文件重新上传联调，确认任务返回 `result_payload.transcription.provider=faster_whisper`
